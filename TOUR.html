1 · The Big Idea

Stodi is not a chatbot. It's a retention loop that owns itself.

Most study tools are reactive: the student opens the app, uploads notes, asks questions — the tool waits. Stodi inverts that:

1

It knows what the student knows — every answer updates a per-topic mastery score (0–100%).

2

It predicts when they'll forget — weaker topics get shorter review intervals (spaced repetition).

3

It texts first — when a topic is due, a personalized quiz lands in the student's Telegram. Nobody opened an app.

Three design pillars:

PillarContrastWhy it matters
Autonomous — the agent initiatesQ&A tools wait for motivated studentsThe students who most need structure never drive
Grounded — official syllabus + marking schemesgeneric LLM contentStodi grades like an examiner; it can't hallucinate the syllabus
Channel-native — lives in Telegram/WhatsAppweb/app-first toolsNo install, data-light, in the app students already open daily

3 · The Five Agents

Each is an ADK Agent(...) — a model + an instruction + tools. The orchestrator holds the other four as sub_agents, and the LLM itself decides who handles each turn (ADK transfer).

🧭 Orchestrator — agents/orchestrator.py

Job: classify intent → route to the right specialist → synthesize a short, chat-friendly reply.

Notable rules: never ask students for internal IDs; route "what packs do you cover?" to Curriculum (MCP registry); keep replies short; ground everything.

No tools of its own — its power is the sub_agents list.

📚 Curriculum — agents/curriculum.py (the source of truth)

Job: own the active Exam Pack. If content didn't come from here, it doesn't exist.

Tools: lookup_topic (fuzzy name→topic), get_past_questions, search_questions (Vertex AI Search when provisioned, local corpus otherwise), get_marking_scheme, get_syllabus_summary, load_pack — plus the MCP toolset (list_packs / pack_manifest / pack_questions, see §7).

The rule that defines the product: "NEVER generate questions or content not in the exam pack."

🧑‍🏫 Tutor — agents/tutor.py (Socratic, relentless)

Job: teach and drill — one question at a time, never hands over the answer, guides the student to it.

Star tool: generate_mcq_drill(topic) — takes the student's plain words ("quadratic equations"), resolves them against the syllabus itself (exact id → name match → word match → highest-weight fallback) and returns real pack questions with answers + schemes for the model to administer. Students are never asked for internal identifiers.

Also: calculate (safe AST calculator — the tutor never does mental math), formula cards, math formatting, study sheets.

🧠 Retention — agents/retention.py (the wedge)

Job: decide what to surface and when. Reads mastery, ranks topics by urgency = (100 − mastery) + hours_since_review, schedules reviews.

Feeds: the morning drip (§5) and "what should I revise today?" queries.

✅ Grader — agents/grader.py (the examiner)

Job: score answers against the pack's official marking scheme. MCQ = deterministic code (grade_mcq). Theory/essay = the LLM applies the scheme point-by-point with partial credit.

Measured: matches examiner-labelled marks exactly on 9/10 held-out answers (§9).

4 · A Message's Journey

One real message, end to end.

student
Quiz me on quadratic equations
1

telegram_bot.handle_text() receives the update, shows "typing…", calls service.handle_message(user_id, text).

2

core/service.py loads the student's pack choice from the store, gets/creates their ADK session, and feeds the message to the Runner.

3

The orchestrator (Gemini) classifies this as a drill request → emits transfer_to_agent(tutor_agent).

4

The tutor calls generate_mcq_drill("quadratic equations") → the tool resolves it to syllabus topic B3 and returns the real pack question x² − 5x + 6 = 0 with its marking scheme.

5

The tutor presents one question (drill protocol) and waits.

6

The student answers → grading (scheme-based, partial credit) → mastery update → reply streams back up through service → bot → Telegram.

stodi
Let's test your knowledge on Quadratic Equations 📚

Solve: x² − 5x + 6 = 0
student
(x−2)(x−3)=0 so x = 2 or 3
stodi
🔥 Perfect! Factored correctly. Ready for the next one? 💪
Failure path by design: if the model call throws (e.g. a 429 quota burst), service.handle_message catches it and replies "Whew, everyone's studying at once! 😅" — a student never sees a 500.

5 · The Autonomous Drip — how Stodi texts first

File: tools/drip_scheduler.py. This is the differentiator.

The spaced-repetition clock

MasteryStatusNext review due
< 50%weak24 h after last review
50–79%medium48 h
80–94%strong72 h
95%+mastered168 h (1 week)

Who gets texted, and what

1

get_students_due_now() scans every profile: any topic past its due time → student is due.

2

build_drip_quiz(student) picks 2 weakest due topics + 1 strong topic ("💪 confidence boost") and pulls real questions from the student's own pack.

3

format_drip_message() renders the "☀️ Good morning!" message.

4

Delivery — two paths: a local poller (background task in the bot, every STODI_DRIP_INTERVAL_MINUTES) or Cloud Scheduler → POST /drip on Cloud Run at 08:00 Africa/Lagos.

Safety latch: STODI_DRIP_DRY_RUN=true makes the scheduler log instead of send — drips can never accidentally message real students from a dev machine.

6 · Exam Packs — the scale story

An exam is content, not code. One JSON bundle per exam/subject in exam_packs/<board>/<subject>/pack.json.

{
  "name": "WAEC Mathematics",  "version": "1.1",
  "syllabus":   { "topics": [ {"id":"B3","name":"Quadratic Equations",
                               "difficulty":"hard","weight":3.0}, …25 ] },
  "past_questions": { "questions": [
      { "id":"WAEC-MATH-2022-02", "topic_id":"B3", "question_type":"theory",
        "question_text":"Solve: x² − 5x + 6 = 0",
        "correct_answer":"x = 2 or x = 3",
        "marking_scheme":"1 mark factorization, 1 mark per root. Total 3.",
        "marks":3 }, …25 ] },
  "retention":  { "max_questions_per_drip":5, "drip_time":"08:00" }
}
25 / 25WAEC Maths — topics / questions
19 / 20WAEC English — incl. essay marking rubrics

Why it scales: a new exam (JAMB ≈ 2M candidates) is a content drop on the same five agents. Each pack is a SKU. The loader (config/exam_pack.py) validates everything through Pydantic — a malformed pack can't load.

7 · MCP — Packs as a Protocol

The Model Context Protocol is a standard way for agents to call external tools. Stodi doesn't just use MCP — it serves it.

1

The server — mcp_server.py (dependency-light, no stodi imports). Exposes three tools over stdio: list_packs, pack_manifest, pack_questions. Runs standalone: python stodi/mcp_server.py.

2

The mount — in agents/curriculum.py, an ADK McpToolset spawns that server as a subprocess and exposes its tools to the agent like any other tool.

3

The point — packs are external, swappable content behind a standard interface. Any MCP-capable agent can mount the same registry. Exam packs as a service.

pack_registry_mcp = McpToolset(
    connection_params=StdioConnectionParams(
        server_params=StdioServerParameters(
            command=sys.executable, args=[str(_MCP_SERVER)]),
        timeout=15.0),
    tool_filter=["list_packs", "pack_manifest", "pack_questions"])

Try it live: ask the bot "What exam packs do you cover?" → curriculum calls list_packs over MCP → answers with exact counts, not vibes.

8 · Memory & Mastery — "we remember you"

Files: tools/agent_tools.py (logic) + persistence/store.py (storage).

The student profile

{ "student_id": "…",
  "current_exam_board": "waec", "current_subject": "mathematics",
  "mastery": { "B3": { "pct": 38, "attempts": 3,
                       "last_review": "2026-06-11T…",
                       "history": [25, 33, 38] } },
  "weak_topics": ["B3","A4"], "strong_topics": ["D1"],
  "total_quizzes": 7, "total_correct": 3 }

The 70/30 rule

A new score never overwrites mastery — it blends: new = 0.7×old + 0.3×latest. One lucky guess can't fake mastery; one bad day can't erase it.

Thresholds

<50% → weak list (drilled hardest) · ≥80% → strong list (used for morale boosters in drips).

10 · Security — the four locks

1 · No inbound bot endpoint

The bot long-polls Telegram, authenticated by the bot token. There is no public webhook to attack. The token lives only in .env (gitignored).

2 · API key gate

On Cloud Run, /chat /switch /progress /drip require X-API-Key (env STODI_API_KEY). A stranger with the URL gets 401, not a model bill. /health stays open.

3 · No eval(), ever

tools/safe_math.py parses expressions into an AST and walks a whitelist. __import__('os') → rejected. Verified in the calculator eval suite.

4 · Blast-radius control

Drips are dry-run gated; evals force the memory store; model 429s degrade to a friendly retry message — never a 500.

12 · Common Pitfalls

Bot answers /start but ignores everything else

Two bot processes are fighting. Telegram allows ONE poller per token; duplicates cause conflicts and messages vanish. pgrep -f telegram_bot must show exactly one PID.

"Reauthentication is needed" / Vertex auth errors

ADC expired: gcloud auth application-default login. The eval harness auto-skips LLM suites (with the reason) instead of failing.

429 RESOURCE_EXHAUSTED

Fresh-project Vertex quota is bursty. The service replies with a friendly retry message and recovers in ~30–60s.

"gemini-3.5-flash 404 NOT_FOUND"

Regional endpoint. Gemini 3.x is served via GOOGLE_CLOUD_LOCATION=global only. Note: deploy.sh uses that variable as the Cloud Run region — keep us-central1 in the deploy shell and set global on the service env.

Cloud Run source deploy: PERMISSION_DENIED on build

New GCP projects don't grant the compute service account build rights. Grant roles/cloudbuild.builds.builder (+ aiplatform.user, datastore.user for runtime) to <project-number>-compute@developer.gserviceaccount.com.

Drip never sends

Checklist: ① STODI_DRIP_DRY_RUN=false? ② does the student have mastery topics at all? ③ is anything actually due (last_review + interval ≤ now)? ④ poller interval short enough to notice?

Bot exits instantly: "TELEGRAM_BOT_TOKEN not set"

load_dotenv() reads .env from the current working directory — launch the bot from inside stodi/.

13 · Test Yourself

Five questions, Stodi-style — click an answer.

Q1. A student types "quiz me on algebra". Which component turns those plain words into a syllabus topic id?

The tool does the resolving (exact id → name → word → weight fallback). Design principle: fix the tool signature, not the prompt.

Q2. Mastery is 100%, then the student bombs a quiz (0%). New mastery?

70/30 weighted blend: 0.7×100 + 0.3×0 = 70. One bad day can't erase real knowledge.

Q3. Why can't a stranger with the Cloud Run URL burn model credits?

All credit-spending routes are key-gated; only /health is open.

Q4. What makes Stodi "autonomous" rather than just another chatbot?

Scheduler fires → retention picks weak due topics → quiz lands on the student's lock screen. Everything else is supporting cast.

Q5. Why does Gemini 3.5 Flash 404 on us-central1?

Set GOOGLE_CLOUD_LOCATION=global for 3.x model ids.
